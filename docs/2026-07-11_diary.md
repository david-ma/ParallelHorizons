First-pass review of the **gallery** repo: a Thalia-hosted 3D WebGL art gallery derived from OwlSketch’s original Met Museum scraper concept, now maintained as David Ma’s exhibition digital twin and a teaching example for CDS2704.

What remains is mostly **productisation**: schema contract, async loading, auth/uploads, persistence, placards, and cleaning up legacy assets and docs.

## Architecture (viewer)

The viewer is well split compared to the original monolithic `main.js`:

```
main.ts          → boot, pointer lock, render loop, physics vs legacy branch
layout.ts        → load floorplan JSON, build walls + artworks
gallery.ts       → hardcoded default scene (30 paintings + one spotlight rig)
artwork.ts       → frames and moulding
spotlight.ts     → GLB fixture + dev sliders + export JSON
movement.ts      → WASD / gamepad, camera-relative velocity
physics.ts       → Rapier floor, walls, player body
types.ts         → Gal, FloorplanBlob
```

**Movement:** WASD and analog stick use camera-relative forward/right on the XZ plane (`movement.ts`). Correct use of `dir × up` for right vector is documented in AGENTS.md.

**Physics:** Rapier initialises async; colliders are built from `wallGroup` meshes after scene creation. Active only when pointer lock is on (`usePhysics = physicsWorld && playerBody && !screensaver`). Screensaver mode keeps legacy manual AABB sliding against walls inside `main.ts`’s render loop.

**Layout modes:**

1. If `GET /gallery-floorplan.json` succeeds → `buildSceneFromFloorplan`
2. Else → `buildDefaultGallery` (hardcoded room, 30 images, featured spotlight)

There is **no `public/gallery-floorplan.json` in the repo** right now (latest commit: “Go back to default floorplan”), so `/view` always uses the default layout unless a file is added manually.

## Architecture (create page)

`/create` is a self-contained floorplan editor:

- 5×5 grid; click cells to activate/deactivate
- Drag photos onto wall bands (north/east/south/west)
- **One photo per wall** in the editor UI
- Export / import / download JSON with `version: 2`
- Charcoal Studio dark palette, scrollable photo sidebar
- D3 v7 loaded from CDN; logic in `gallery_creation.ts`

Mock catalog: 12 photos pointing at `/img/Artworks/0.jpg` … `11.jpg`.

## JSON schema — current contract (informal)

Editor exports (`gallery_creation.ts`):

```json
{
  "version": 2,
  "grid": { "rows": 5, "cols": 5 },
  "activeCells": ["2,2"],
  "placements": {
    "2,2": { "north": "photo-0", "east": "", "south": "", "west": "" }
  },
  "photoCatalog": [{ "id": "photo-0", "title": "Photo 1", "src": "/img/Artworks/0.jpg" }]
}
```

Viewer consumes (`types.ts` / `layout.ts`):

- Same shape, but **viewer accepts `string | string[]` per wall** and can place **multiple artworks per wall** with horizontal spacing
- Backward compat: whole-cell string or legacy shapes normalised on import in the editor
- `photoCatalog[].title` is typed but **not rendered** (placards not implemented)

**Gap:** AGENTS.md changelog mentions “multi-photo per cell” on the viewer side, but the editor was simplified to one photo per wall. The schema is not formally documented or version-validated beyond the editor’s `version: 2` field.

## Session notes — 2026-07-11 (floorplan editor UX)

### `/create` editor: drag-move + undo/redo

**Problem:** Deactivating a cell deletes all artworks on that cell with no recovery. Placed thumbnails could not be repositioned on the grid except by re-dragging from the sidebar.

**Planned behaviour:**

| Action | How |
|--------|-----|
| Move artwork on grid | Drag a placed thumbnail on `#floorplan-svg` to another wall band (same or different cell). Swaps if the target wall already has a photo. |
| Drag from sidebar | Unchanged — drop onto a wall band to place (copy). |
| Undo | `Ctrl+Z` / `Cmd+Z` — cell activate/deactivate, placements, moves, placard snapshot |
| Redo | `Ctrl+Y` or `Ctrl+Shift+Z` / `Cmd+Shift+Z` |
| Shortcuts disabled | While focus is in a placard `<input>` (so typing isn't intercepted) |

**Implementation approach:**

- Snapshot editor state (`activeCells`, `placements`, `photoCatalog`, selection) before each mutating action.
- `application/x-gallery-from` drag payload marks moves from the grid vs new drops from the photo list.
- History stack capped (~100 entries); import/load/reset wrapped in `mutate()` so they are undoable.

**Shipped:** `gallery_creation.ts` — `mutate()` / `undo()` / `redo()`, draggable `.placed-art` SVG images with `application/x-gallery-from` payload, wall-drop swap logic, placard edit batches one undo step per focus session.

## `/create` UX ideas (backlog)

The editor works but still feels like a dev tool: cramped sidebar, tiny grid thumbnails, hidden affordances, and a manual export → save → view loop. Ideas below, roughly ordered by impact vs effort.

### High impact — layout & safety

1. **Confirm destructive actions** — Deactivating a cell or Reset should show a small confirm dialog (“Remove 4 artworks from this room?”). Undo helps, but prevention is better UX than recovery.

2. **Separate “remove artwork” from “deactivate cell”** — Clicking a cell toggles the whole room off. Better: right-click cell → Deactivate room; click placed art → select; **Backspace** or trash icon → remove from wall only. Keeps layout editing from feeling like a trap.

3. **Bigger placed thumbnails + wall affordances** — 14px thumbs are hard to grab. Show ~32px previews on walls, faint “N/E/S/W” labels on bands, and a stronger hover state so drop targets are obvious before drag starts.

4. **“Save & preview” one click** — POST or write `gallery-floorplan.json` server-side (Thalia route), then open `/view` in a new tab. Removes the export/download/copy ritual entirely.

5. **Tabbed or accordion sidebar** — Today: controls + raw JSON + hint + placard + photos in one scroll column. Split into **Layout | Photos | Labels** tabs so the grid gets more horizontal space and JSON moves behind “Advanced”.

### Medium impact — clarity & feedback

6. **Placed vs unplaced photos** — Grey out or badge photos already on a wall; filter “Unplaced only” so the library doesn’t feel endless when you have 12 mocks.

7. **Selection model on the grid** — Single-click thumbnail = select (placard panel updates); double-click = edit placard. Cell background click = select room, not instant deactivate (require second action or modifier key).

8. **Undo/redo UI** — Toolbar buttons + disabled state when stack empty; brief toast (“Undid: deactivated cell 3,1”) so keyboard shortcuts are discoverable.

9. **Live minimap or 3D peek** — Small embedded `/view` iframe or static isometric preview that updates on change (debounced). Helps users who don’t think in top-down grid coordinates.

10. **Drag ghost** — Custom drag image (full photo tile) instead of default browser ghost; show valid drop zones highlighted for the duration of the drag.

11. **Empty-state onboarding** — First visit: “1. Click cells to add rooms → 2. Drag photos to dashed edges → 3. Save & preview”. Dismissible; link to schema doc.

### Polish & power-user

12. **Paint tool for cells** — Brush mode: click-drag to activate/deactivate multiple cells instead of one-by-one toggling (with undo batches).

13. **Context menu on placements** — Right-click artwork on grid: Edit label, Remove, Swap with…, Jump to photo in library.

14. **Keyboard shortcuts cheat sheet** — `?` opens modal: Z/Y undo, Delete remove art, Esc clear selection.

15. **Collapsible JSON panel** — Default hidden; devs expand when needed. Frees ~180px vertical space for photos/placards.

16. **Rename route & homepage link** — `/create` → “Edit gallery” on `/`; drop “(Dev)” from title once save/preview exists.

17. **Mobile/tablet** — Touch drag on SVG is flaky; consider simplified flow (list of rooms → pick wall → pick photo) for narrow screens.

18. **Upload photos in-editor** — UploadThing picker in Photos tab; auto-add to `photoCatalog` instead of 12 hardcoded mocks.

### Longer term (product)

19. **User galleries** — Edit page at `/edit/:id` loads from DB; autosave; share link for read-only `/view/:slug`.

20. **In-world editing** — Pointer-lock in `/view`, aim at artwork, press E to edit placard (connect editor to 3D context).

### Suggested next slice (if picking one)

**#4 Save & preview + #5 tabbed sidebar + #3 bigger thumbs** — biggest perceived jump from “dev JSON toy” to “gallery editor” without a full rewrite.

## Performance, memory & telemetry

### What we know today (2026-07-11)

The viewer is lightweight by modern WebGL standards, but we have **almost no instrumentation** beyond a live FPS counter. This section records a first-pass estimate and a plan to capture real data.

#### Texture / RAM pressure (wall & ceiling change)

Procedural wall and ceiling textures (`materials.ts`) are two **512×512 RGBA** canvases:

| Asset | JS heap (approx.) | GPU VRAM (approx.) |
|-------|-------------------|---------------------|
| Wall source texture | ~1 MB | ~1 MB |
| Ceiling source texture | ~1 MB | ~1 MB |
| **Incremental cost vs flat white walls** | **~2 MB** | **~2 MB** |

That is negligible compared to the rest of the scene:

| Asset class | Notes |
|-------------|-------|
| Artwork JPGs | Dominant cost. Current floorplan has ~9 placements; each decoded GPU texture is often **1–4 MB** depending on resolution. Catalog on disk ~4 MB total; GPU decode can exceed file size. |
| Floor | `Floor.jpg` (512×512) — already loaded before wall/ceiling work. |
| Spotlight GLB | ~30 KB on disk, but `GLTFLoader.load()` is called **once per artwork** — redundant parses/clones, not yet cached. |
| Placards | One small `CanvasTexture` per labelled artwork. |
| Rapier | WASM physics heap on top of the above. |

JSON layout shares **one wall material** across all wall segments (good). Default hardcoded gallery creates four wall material instances but they reference the same canvas source.

#### FPS

Computed in the render loop (`main.ts` → `framerate()`), displayed in `#devonly`:

```
FPS: <span id="fps"></span>
```

Updated every ~15 frames. A **duplicate** updater exists in `gamepadtest.ts` (jQuery, every 30 frames when a gamepad is connected).

There is **no shadow mapping**, which keeps GPU cost down. Per-artwork `SpotLight` count scales with placement count (currently ~9 in sample floorplan). Lambert/Phong materials, modest geometry — **60 FPS is the desktop target**, but we are not recording proof.

#### What we are *not* tracking

| Metric | Status |
|--------|--------|
| FPS history / percentiles | Not captured |
| RAM / JS heap | No `performance.memory`, no logging |
| GPU / texture memory | No `renderer.info` |
| Draw calls / triangles | No Three.js `Stats` panel |
| Load timings (JSON, textures, GLB, Rapier init) | Not captured |
| Persistence / analytics | Nothing sent or stored |

The diary and AGENTS.md note that `main.ts` still mixes framerate counting into the render loop — a refactor candidate, and a natural hook for a future `telemetry.ts` module.

### Observed browser differences (informal)

Subjective testing on the same machine and gallery layout:

| Browser | Notes |
|---------|-------|
| **Cursor built-in browser** | Noticeably **faster and smoother** — snappier pointer-lock feel, cleaner frame pacing during walk-through. |
| **Chrome** | Usable but feels heavier; more jank when moving through the corridor. |
| **Safari** | **Slowest** of the three tested — lower perceived FPS, less responsive movement. |

We do not yet have numbers to back this up. Likely contributors (hypotheses until instrumented):

- Different WebGL backends and compositor behaviour (Safari / Metal vs Chrome / ANGLE).
- Cursor's embedded browser may run with different GPU process isolation or vsync defaults.
- Spotlight count + texture upload path may hit Safari's stricter memory or shader limits sooner.
- No adaptive quality tier — same scene graph and light count everywhere.

**Action:** Treat browser/engine as a first-class dimension in telemetry, not an afterthought.

### Telemetry plan (future)

Goal: lightweight, privacy-conscious client telemetry so we can answer “how fast? how much RAM? on what setup?” without guessing.

#### Phase 1 — Dev overlay (localhost only)

Extend `#devonly` (or a `?debug=1` query param) with a collapsible panel:

| Field | Source |
|-------|--------|
| FPS (current + 1 s / 10 s avg) | Existing `framerate()` + rolling buffer |
| Frame time ms | `performance.now()` delta |
| Draw calls, triangles, textures, geometries | `renderer.info.render` / `renderer.info.memory` |
| JS heap used / total | `performance.memory` (Chrome/Edge; graceful omit elsewhere) |
| Scene counts | Artworks, walls, spotlights, colliders (from `gal`) |
| Load timings | Floorplan fetch, Rapier init, first frame rendered |

No network; on-screen only. Replaces duplicate FPS logic in `gamepadtest.ts`.

#### Phase 3 — Product analytics (when auth exists)

- A/B: procedural vs flat walls, spotlight-per-art vs shared rig.

#### Client environment fields to capture

| Category | Fields |
|----------|--------|
| **Display** | `window.innerWidth/Height`, `screen.width/height`, `devicePixelRatio`, `orientation`, fullscreen state |
| **Browser** | Parsed name/version from UA (or `navigator.userAgentData` where available), engine, mobile vs desktop heuristic |
| **System** | `navigator.platform`, `navigator.hardwareConcurrency`, `navigator.deviceMemory` (Chrome), `navigator.language`, `Intl` timezone |
| **WebGL** | `WEBGL_debug_renderer_info` vendor/renderer (with user consent note in privacy policy), max texture size, max vertex uniforms |
| **Capabilities** | Pointer lock, gamepad present, WebGL2 vs WebGL1, `OffscreenCanvas` support |
| **Network (optional)** | `navigator.connection.effectiveType` if available — useful for texture loading strategy |

#### Privacy & storage

- Default **opt-in** or anonymous-only until privacy policy covers it.
- Do not log raw IP in application payload; let reverse proxy handle geo if needed.
- Hash or truncate `userAgent` if storing long-term; keep parsed browser/OS columns for aggregation.
- Retention: 90 days for raw events; roll up to daily medians per browser × artwork-count bucket.

#### Immediate takeaway

The wall/ceiling textures are **not** the performance story. **Artwork texture count, per-artwork spotlights, browser WebGL stack, and uncached GLB loads** are the levers worth measuring — and we currently have no data pipeline for any of it. Cursor's browser feeling best and Safari worst is a strong signal to prioritise cross-browser snapshots before optimising blindly.

**Update (same day):** Chrome's ~30 FPS cap on the same machine was traced to **Settings → Performance → Energy Saver**, not WebGL or our code. `chrome://gpu` showed hardware acceleration and a 120 Hz panel while rAF was still throttled. With Energy Saver off, Chrome matches expected ProMotion pacing.

## 2026-07-12 — SmugMug URLs, thumbnails, WebGL, Monetise `/mirror/`

End-to-end upload and display debugging after first real SmugMug photos landed in the BINGO album.

### UploadThing client bug (photos stuck in UT only)

**Symptom:** Server log showed `POST /api/uploadthing` but never `POST /uploadPhoto`; files in UploadThing, nothing in DB or SmugMug.

**Cause:** UploadThing v7 client returns **`ufsUrl`**, not `url`. `photo-upload-client.ts` only checked `item.url`, skipped every file, and swallowed errors.

**Fix:** Resolve `ufsUrl ?? url ?? appUrl`; normalize nested `{ data: … }`; POST `/uploadPhoto` with `uploadThingUrl`, `fileKey`, `filename`, `size`, `title`. Surface failures in the editor/library toasts.

### SmugMug URL shape (thumbnails vs 3D)

SmugMug upload ack **`Image.URL`** is often a **gallery page** (`https://photos.david-ma.net/…`), not a JPEG. Thumbnails come from **`ThumbnailUrl`** on the verbosity `AlbumImage` (`photos.smugmug.com/…/Th/…`).

| Surface | URL used |
|---------|----------|
| `/library`, `/create` sidebar | `thumbnailUrl` (or `src` fallback) |
| DB `photos.url` / floorplan `src` | Large CDN URL (`…/L/…-L.jpg`) via `config/smugmug-urls.ts` |
| Legacy rows with page URL in `url` | `resolvePhotoDisplaySrc()` upgrades on read in `photo-store.ts` |

New uploads store correct CDN URLs at insert time; floorplan merge in the editor prefers API URLs over stale saved `src`.

### WebGL black artwork (CORS)

**Symptom:** Thumbnails fine in library/create; **black frames** in `/view` with valid SmugMug `-L.jpg` in floorplan JSON.

**Cause:** SmugMug CDN does **not** send `Access-Control-Allow-Origin`. `<img>` works; Three.js `TextureLoader` needs CORS for WebGL textures. SmugMug API [method overrides](https://api.smugmug.com/api/v2/doc/advanced/overrides.html) apply to JSON API requests only — no CDN CORS knob.

**Fix:** Route cross-origin artwork through **[Monetise](https://monetiseyourwebsite.com) `/mirror/`** — passthrough fetch + `Access-Control-Allow-Origin: *`. Implemented in `/usr/local/dev/monetise` (`config/mirror.ts`); gallery does **not** host its own image proxy.

Gallery wiring:

- **`MONETISE_MIRROR_ORIGIN`** in `.env` (e.g. `https://monetiseyourwebsite.com`)
- `loadMirrorOrigin()` → `gallery.hbs` → `globalThis.GALLERY_MIRROR_ORIGIN`
- `src/js/artwork-source.ts` rewrites external floorplan `src` to `{origin}/mirror/{url}`
- `layout.ts`: `TextureLoader.crossOrigin = 'anonymous'`

Verified live: mirror returns 200 + JPEG + CORS; `/view/untitled-gallery-3` artwork renders when env is set.

### Config reference

```env
# .env — required for 3D SmugMug artwork
MONETISE_MIRROR_ORIGIN=https://monetiseyourwebsite.com
```

Local Monetise dev: `cd /usr/local/dev/monetise && bun run dev`, then set origin to that host/port.
